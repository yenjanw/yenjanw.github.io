網站載入應是有再快一點

載入首頁居然高達8.7MB#

其實網站也沒啥人在看，SEO也普普，也沒有什麼吸引人的人章，這裡只是我實作BLOG的一個場所而已，但是我對效率與使用者經驗還是很在乎的。

前幾天，利用pingdom測試了一下網站的載入需要多久的時間，在Google Cloud Platform的加持下，速度是還蠻不錯的。從美國San Jose載入網站只要四秒鐘左右，但是載入的容量嚇到我了，沒啥內容的網站居然高達8.7MB(真的是87)。讓我重新再審視一下到底是哪裡出錯了。

開始網站優化#

整個網站都是用HEXO產生出靜態的html檔，然後再上傳到我的GCP之中，開始查看有沒有相關的Plugin可以幫助我將生成的容量減少。

第一步:減少圖檔的載入#

每一個頁面，載入的所需時間是不相同的，最在意是首頁的部份，怎麼會需要高達8.7MB呢?

原來是我沒有做摘要，就是因為沒有摘要的因素，所以首頁幾乎是把我所有的文章都讀了出來，相當然圖檔也全讀出來。這解決方法有三種。

  1. 在文章的front-mater中加入description的描述，也就是在YAML之中寫入文章的摘要，這樣生成文章就會有簡單的描述文字只顯示在首頁之中。

  2. 利用主題NEXT的功能，在/themes/[主題]/_config.yml之中加入:

    auto_excerpt: enable: true length: 150

    這樣就會自動生成150字的摘要，若要更多的字數，可自行更改。

  3. 在文章之中加入<!-- more -->的字串，如此生成HTML時，就會知道哪些是摘要，哪些是內文。最大好處是可以自己控制文章的摘要，不用被固定的字數所限制。

因為我的文章還不多，畢竟是還在與GCP之間的磨合期，所以改每篇文章其實還好，我就利用這方式，將每個文章都加入了摘要字串。

光是利用這方式，載入已由8.7MB下降到2.2MB，功效不可謂不大。

第二步:壓縮程式碼#

其實在HEXO plugin的部份有很多的可以壓縮的方式說明，我最後選定的是neat的方式進行，因為可以有參數作設定。

  1. 首先安裝hexo-neat到系統中

    npm install hero-neat  —save

  2. 在hexo下的配置，將以下屬性輸入

    hexo-neat

neat_enable: true

neat_html:
  enable: true
  exclude:

neat_css:
  enable: true
  exclude:
- '*.min.css'

neat_js:
  enable: true
  mangle: true
  output:
  compress:
  exclude:
- '*.min.js'

    其中可以針對html、css、js分別作設定，但我執行js的部份就是會一直出現執行過久的狀況(畢竟我選的是最低階的伺服器)，在省錢第一的原則下，又不想太麻煩，所以就把js的部份取消了。

    經過這一步驟，載入又降到1.7MB了，個人已經覺得算是滿意。

接下來呢?#

其實程式碼以現在的網路速度而言，真的都不算什麼，最大的資料來源是圖片。之前的部份我就真的懶得再動手處理，往後的圖片我想我都會經過壓縮處理再上傳吧，不然真得很佔空間，且效率也不高。
