GCP最低配置不適合HEXO壓縮

差一點就當機了 利用Google PageSpeed Insight之中的服務，了解自己網站還有哪些是可以改善的部份，一直提示我網站的java script、CSS、HTML沒有做壓縮的動作，建議我執行。 我重新查了HEXO Plugin的內容，找到了上述三種的壓縮方式，利用之後就gg了! 超高的系統使用率 因為壓縮需要用到系統的CPU與HD交互運算，大家都知道，為了省錢，我所選擇的是GCP最低配置。當我一執行hexo內容更新時，哇! 如下圖: CPU使用率好像還好，但是HD使用率卻是高的嚇人。然後就一直出現錯誤的訊息，網站也不容易進入，最慘的是連SSH都進不去。好幾次我想要重新開機就好了，但是一直忍住。 看著HD讀取24M/s的使用，就知道一定是程式發生錯誤，造成系統這麼高的使用量(有高嗎?)。 刪除壓縮功能 好不容易SSH進入系統，把相關的minifier全都uninstall掉 ​ npm uninstall hexo-XXXX-minifier 執行完後，系統終於回覆正常了。 這經驗告訴我，若是要做JS、CSS、HTML壓縮的工作，建議拉回本機端來做。GCP最低配置不適合作壓縮的工作，真的。

純文字的力量!

又完成了一次網站移轉 你是有病哦，一直把網站換來換去，這所謂_程式員就是愛折騰_。這一次其實也不是什麼搬家的行為，只是因為之前網站一直被警告，讓我非常的納悶。我blog來看的小貓沒有兩三隻，為何一直被告知使用量過高的問題。其實追根究底還不就是因為Wordpress這個世界使用率超過三成的CMS(Content Management System)，在生成文章時的資源消耗太多了。有很多嗎?其實也不高。畢竟我選擇的是最低階的配置造成資源不夠使用的冏境。 從我之前的文章可以看到，最近我一直在使用markdown的語法來作文。其實markdown與html語法只有一線之隔而已，從純文字就可以轉換成網頁的呈現型式，真的是好用的語法。現在你所看到的文字，全是用markdown語法寫出來的。 網站架構 這BLOG是用HEXO所架出來的，不過我跟大家不一樣的地方是，編譯我放在GCP雲端上執行。為何不在近端執行就好了呢? 因為我電腦的變動性，說實話會比伺服器還快得多，哪天資料全不見了，會欲哭無淚哦。 所以現在寫BLOG的流程是: 用typora與markdown語法寫文章。 用Filezilla上傳到GCP。 從GCP執行編譯備份到Github上。 再從GitHub備份到我的GCP之中。 好處是我的文件在三個地方都有備份，一份在電腦、一份在GitHub、一份在GCP，所以假如文章再會不見，那應該是火星人進攻地球的時候才會發生吧。 新版網站的優點 版面更加的簡單(其實以前就已經很簡單了)。 右側邊欄有TOC(Table of Content)。 直接點選TOC可以直接跳到文章的標題。 更神奇的是TOC會隨著文章卷動出現反白。 SEO好像做得更好了。 全純文字，不怕遇到駭客，假如要能駭進來，那要從Linux的底層進攻。安全性更高。 完全沒有資料庫作底層運算，所以不用擔心效能問題。 所以這個版本，一次解決了我多個問題。往後我還是一樣，以簡單為出發點。 純文字有何種力量? 說了老半天文不對題，其實我就是想說明純文字之後，到底有何改變? 寫文章時不用再費心編排格式。用markdown寫完之後，直接編譯成HTML格式，快速用方便。 文章的格式簡單、條理鮮明，利用標題與排列的方式，簡單就能表達出作者的文意。 容易閱讀，這是我最喜愛的部份。有些BLOG會做過多的修飾，讓多媒體的呈現更豐富，但那些東西其實只會使讀者分心。純文字化之後，沒有過多干擾，相對之下比較不容易分心。 假如在Word上寫文章，複製出來的文字常常都帶有內碼。沒經驗的說可以試著用Word寫段話，再轉貼到outlook去試看看，通常會連內碼都帶過去了。假如對方不用outlook來看的話，發生編碼錯誤的機率很高。 純文字之旅 從Wordpress移轉到HEXO，本身就是一個很大的挑戰。其實我不是科班出身的，對於command line的操作能力也只是一般般而已，但是看到別人網站上受可移動的TOC一見傾心，下定決心一定要讓自己的網站與Wordpress網站有那不一樣的功能，就開始折騰了。花了三天研究該如何進行，又花了一個晚上把程式搞定滿再花一個晚上把文章搬家，我想以後應該不會再想這樣搞了吧! 還是乖乖的應用，留下有用的資料在茫茫的網海之中。

不要怪我用奇怪的圖作封面，我感覺被Google嘲笑了 開站一天而已，登入到雲端後台，發現出現下圖 再點進去看 這樣的意思是我硬體消耗太多了嗎?，哈哈! 實際上從系統上顯示的CPU使用率而言，最高不過也才25%而已。 所以決定不去理會這樣的訊息。這是因為若調整成Google所建議的配置，一個月就差不多快一千元台幣的費用了，那一年不就破萬。 這網站不太想以營利為目的，所以先維持原配置，再視狀況而定。

網站載入應是有再快一點

載入首頁居然高達8.7MB 其實網站也沒啥人在看，SEO也普普，也沒有什麼吸引人的人章，這裡只是我實作BLOG的一個場所而已，但是我對效率與使用者經驗還是很在乎的。 前幾天，利用pingdom測試了一下網站的載入需要多久的時間，在Google Cloud Platform的加持下，速度是還蠻不錯的。從美國San Jose載入網站只要四秒鐘左右，但是載入的容量嚇到我了，沒啥內容的網站居然高達8.7MB(真的是87)。讓我重新再審視一下到底是哪裡出錯了。 開始網站優化 整個網站都是用HEXO產生出靜態的html檔，然後再上傳到我的GCP之中，開始查看有沒有相關的Plugin可以幫助我將生成的容量減少。 先說明我現有的配置 "hexo": "^3.2.0", "hexo-deployer-git": "^0.2.0", "hexo-generator-amp": "^2.0.12", "hexo-generator-archive": "^0.1.4", "hexo-generator-category": "^0.1.3", "hexo-generator-feed": "^1.2.0", "hexo-generator-index": "^0.2.0", "hexo-generator-search": "^2.0.0", "hexo-generator-searchdb": "^1.0.3", "hexo-generator-seo-friendly-sitemap": "0.0.20", "hexo-generator-tag": "^0.2.0", "hexo-imgur": "0.0.2", "hexo-server": "^0.2.0", "minimatch": "^3.0.3" 第一步:減少圖檔的載入 每一個頁面，載入的所需時間是不相同的，最在意是首頁的部份，怎麼會需要高達8.7MB呢? 原來是我沒有做摘要，就是因為沒有摘要的因素，所以首頁幾乎是把我所有的文章都讀了出來，相當然圖檔也全讀出來。這解決方法有三種。 在文章的front-mater中加入description的描述，也就是在YAML之中寫入文章的摘要，這樣生成文章就會有簡單的描述文字只顯示在首頁之中。 利用主題NEXT的功能，在/themes/[主題]/_config.yml之中加入: auto_excerpt: enable: true length: 150 這樣就會自動生成150字的摘要，若要更多的字數，可自行更改。 在文章之中加入<!-- more -->的字串，如此生成HTML時，就會知道哪些是摘要，哪些是內文。最大好處是可以自己控制文章的摘要，不用被固定的字數所限制。 因為我的文章還不多，畢竟是還在與GCP之間的磨合期，所以改每篇文章其實還好，我就利用這方式，將每個文章都加入了摘要字串。 光是利用這方式，載入已由8.7MB下降到2.2MB，功效不可謂不大。 第二步:壓縮程式碼 其實在HEXO plugin的部份有很多的可以壓縮的方式說明，我最後選定的是neat的方式進行，因為可以有參數作設定。 首先安裝hexo-neat到系統中 npm install hero-neat —save 在hexo下的配置，將以下屬性輸入 hexo-neat neat_enable: true neat_html: enable: true exclude: neat_css: enable: true exclude: - '*.min.css' neat_js: enable: true mangle: true output: compress: exclude: - '*.min.js' 其中可以針對html、css、js分別作設定，但我執行js的部份就是會一直出現執行過久的狀況(畢竟我選的是最低階的伺服器)，在省錢第一的原則下，又不想太麻煩，所以就把js的部份取消了。 ...

網頁不適合透過行動裝置瀏覽

經過一番奮鬥，我的網站終於可以在Google上找到了，但是發現有一項特別您的網頁不適合使用行動裝置瀏覽，針對這一點我覺得莫名奇妙 因為我特別在Wordpress挑上RWD的版型，且用手機看都很正常啊，怎麼會到Google之後反而發生了問題。為了這件事情，特別花一點時間往內查詢錯誤到底在哪裡? 在Google相容性測試的檢查 由檢查可以看到，有提到23資源遭到封鎖，再點選進去看是哪23項資源，幾乎都是與版型相關的CSS設定，那當然不符合啦。 相容性檢查的呈現頁面 在手機上呈現都沒有問題 找到問題的原因 針對封鎖的資源查詢可能發生的原因，依照Google說明頁面找到的說明 這裡的資源指的是頁面所包含的外部資源，例如圖片、CSS 或指令碼檔案。如果 robots.txt 檔案禁止 Googlebot 存取資源，測試結果頁面就會顯示「封鎖的資源」警告。視遭封鎖的資源而定，這可能會對網頁帶來很大的影響。舉例來說，如果有一張大型圖片遭到封鎖，儘管頁面看似適合透過行動裝置瀏覽，實際上卻可能不是如此；如果遭到封鎖的是 CSS 檔案，則可能會導致網頁套用不正確的字型樣式 (例如字體太小，不適合在行動裝置上閱讀)。這類情況會影響到行動裝置可用性得分和 Google 檢索您網頁的能力，因此您必須確保 robots.txt 檔案不會禁止 Googlebot 存取重要資源。 因為Google並不是真的去我的網頁看網頁，只是派個Googlebot機器人去查而已，而這機器人會因為網站設定的robots.txt檔所限制它所能查看的資料，所以假如限制住了，當然就說版型不合啦。(說實話，還蠻笨的) 所以我重新修改robots.txt之後，一切就正常了。也順便提交說明網頁已修改，請機器人再次來爬。 為何要讓行動裝置可以瀏覽呢? 為的就是SEO(Search Engine Optimization)，因為以現在Google的策略是行動優先，未來假如沒有做好行動優化的頁面，基本上都會往後排名。像我那不適合行動的頁面，搜尋到已經是第三頁了，往過往的經驗，只有13%的人會按到第三頁，這就可以知道為何要爭第一頁了。 並不是我的文章多有價值，一定要人找得到。那只是一種奇蒙子的問題而已
